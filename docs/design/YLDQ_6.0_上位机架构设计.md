# YLDQ 6.0 上位机架构设计

## 1. 设计结论

新版上位机定位为“YLDQ 6.0 远程监控、控制、诊断和历史趋势工作台”，下位机当前 Modbus V7 实现是唯一协议真相，不保留旧协议、旧点表或旧数据格式的兼容分支。

核心原则：

1. 只支持协议字 `0x0700`，连接后第一条业务请求必须读取 FC04 `0..9` 并校验协议字；不匹配时停止采集，不猜测协议版本。
2. 只实现当前固件开放的 FC02、FC03、FC04、FC06、FC16；不再展示或发送已被固件拒绝的 FC05、FC15。
3. 串口通信只支持当前下位机实际使用的 Modbus RTU 通道；删除旧上位机中的 Modbus TCP 入口。
4. 请求和响应都直接使用标准 Modbus RTU ADU，不使用 SLIP 或其他私有外层封装，保证上位机和通用 Modbus RTU 工具可直接通信。
5. 三路温湿度、压力、流量、呼吸状态、加热输出、三路阀门、告警、通信健康是一级可视化对象。
6. 配置数据与即时控制严格分离。配置采用“暂存—校验—提交/放弃”事务；即时控制写入后立即回读。
7. 呼吸流量和呼吸状态作为真实工况保留；删除呼吸预测、参数推荐、参数模拟等上位机推测逻辑。
8. 历史曲线保留并增强，但历史存储改为新版独立数据模型，不继续伪装成旧下位机 CSV 格式。
9. 协议点表只在后端维护一份，前端通过接口获取显示元数据，禁止 Python 和 JavaScript 各复制一套寄存器定义。

## 2. 当前事实基线

设计依据：

- `SerialLinker/docs/MODBUS_V7_PROTOCOL.md`
- `SerialLinker/protocols/modbusModule.c`
- `SerialLinker/protocols/modbus_ext.c`
- `App/Inc/config_v6.h`
- `App/Inc/appErr.h`
- `devices/Inc/ValveActuator.h`
- `devices/Inc/ValveControl.h`
- `SerialLinker/core/LinkerDriver.c`

下位机当前能力摘要：

| 类型 | 当前能力 |
| --- | --- |
| 协议 | Modbus V7，协议字 `0x0700` |
| 地址规则 | 零基地址，F32/U32/U64 高字在前 |
| 读取 | FC02、FC03、FC04 |
| 写入 | FC06、FC16 |
| 扩展 | 功能码 `0x66`，首版上位机不依赖它 |
| 物理链路 | 标准串口 Modbus RTU，无额外封装 |
| 温湿度 | T1/左、T2/右、T3/上共三路 |
| 过程量 | 压力、流量、呼吸状态 |
| 输出 | HTC1、HTC2、防冻、报警输出 |
| 阀门 | 上、左、右三路，含位置、故障、电流、控制源和远程租约 |
| 告警 | 三个 U32 错误组，位号对应 `ErrorPosition` |
| 配置写入 | 写保持寄存器形成暂存，`HR3=0xC6A6` 提交，`HR3=0xD15C` 放弃 |

## 3. 产品功能范围

### 3.1 必须提供

1. 设备档案和串口连接。
2. 多设备监控；同一串口上的多个从站必须串行调度，禁止并发占用串口。
3. 三路温湿度独立显示、质量状态、有效性和趋势曲线。
4. 压力、压力类型、流量和呼吸状态显示。
5. HTC1、HTC2、防冻和报警输出的实际状态、自动/强制模式及累计启动次数。
6. 上、左、右阀门的显示状态、执行状态、位置、故障原因、电流 ADC、控制源和远程控制剩余时间。
7. 全量错误位图解析、当前告警、恢复记录和告警时间轴。
8. 配置读取、编辑、差异预览、提交、放弃、结果确认。
9. 即时控制：远程加热、三路输出模式、三路阀门目标、阀门故障复位。
10. 实时曲线、历史查询、事件叠加、CSV 导出和截图。
11. 通信诊断：请求/响应、CRC、超时、异常码、连续失败、最近成功/失败时间。
12. 本地记录和会话管理。

### 3.2 明确删除

1. 旧 Modbus 点表以及旧线圈、旧离散输入、旧任务寄存器模型。
2. Modbus TCP 配置和通信代码。
3. 旧版 CSV 目录导入与离线分析主流程。
4. 客户交付静态报告生成流程。
5. 参数推荐、呼吸预测、参数模拟和相关 UI。
6. 将上位机采样伪装为下位机 `breath_data` 或 `run` 日志的兼容桥。
7. 在前端复制寄存器地址、数据类型和枚举的做法。
8. 通用“任意十六进制自动轮询”作为普通用户功能；仅在诊断页保留受控的单次调试发送。
9. 下位机未实现的软件复位、线圈控制和定时任务编辑入口。

### 3.3 保留并重构

1. ECharts 及其缩放、平移、框选、图例、tooltip、全屏等友好曲线交互。
2. 多设备档案和设备切换思路。
3. 串口通信日志视图，但改成 V7 帧语义。
4. 本地打包和一键启动能力。
5. 可复用的颜色、卡片、布局等视觉资产；业务页面和状态模型重写。

## 4. 总体架构

```text
浏览器界面
  ├─ REST：设备、配置、控制、历史查询
  └─ SSE：实时快照、告警、连接状态、控制结果
          ↓
本地 Python 服务
  ├─ API 层
  ├─ 设备与采集服务
  ├─ 配置事务服务
  ├─ 控制与租约服务
  ├─ 告警/事件服务
  ├─ 历史查询服务
  └─ 协议 V7 领域层
          ↓
端口调度器（每个 COM 口唯一）
          ↓
SerialTransport → Modbus RTU ADU（含 CRC）
          ↓
YLDQ 6.0 下位机

采集/事件 → 单写入队列 → SQLite → 历史曲线/CSV 导出
```

### 4.1 技术选型

- 后端：Python 3，业务代码使用标准库；串口依赖 `pyserial`。
- HTTP：保留本地服务模式，改用 `ThreadingHTTPServer` 承载 REST 与 SSE。
- 前端：原生 ES Modules + ECharts，不引入大型前端框架。
- 存储：SQLite，使用 Python 标准库 `sqlite3`；CSV 仅作为导出格式。
- 打包：PyInstaller，保留一个主工作台入口和一个 spec。

选择该方案的原因是部署简单、离线可运行、便于打包，同时能彻底拆开当前过大的 `dashboard_server.py` 和 `app.js`。

### 5.1 `protocol_v7/catalog.py`

V7 点表的唯一事实源。每个点定义：

- `id`、中文名称、区域、地址、字数、数据类型、单位。
- 读写权限、轮询组、枚举映射、无效值规则。
- 配置分组、值域、精度、危险级别。
- UI 展示分组与历史记录策略。

前端不得硬编码地址。`GET /api/v1/schema` 返回经过裁剪的展示元数据。

### 5.2 `protocol_v7/codec.py`

只负责协议值转换：

- U16、BOOL、F32、U32、U64。
- 高字在前的多寄存器拼接。
- `0xFFFF` 不可用值。
- 设备状态、传感器状态、阀门状态、故障原因、控制源等枚举。
- 三个 U32 告警组的位展开。

所有解析失败必须产出“数据质量异常”，禁止用 0 代替坏数据。

### 5.3 `serial_transport.py`

职责：

1. 生成 RTU ADU 并附加 CRC16。
2. 直接发送标准 RTU ADU，不增加任何私有封装。
3. 串口读写、超时和基于响应功能码/字节数的判帧。
4. 校验响应地址、功能码、异常码、长度和 CRC。
5. 保留原始 RTU 帧和解析结果供诊断页查看。

### 5.4 `port_scheduler.py`

每个物理串口只创建一个调度器和一个串口对象：

- 同一端口上的不同从站按轮询计划轮转。
- 一个请求完成或超时后才能发下一请求。
- 控制命令优先于普通轮询，但不能打断正在进行的帧。
- 连接状态分为：未连接、连接中、在线、降级、离线。
- 连续失败采用有限退避，恢复后立即读取系统摘要。

### 5.5 `acquisition_service.py`

将寄存器块解码为统一 `DeviceSnapshot`：

```text
DeviceSnapshot
  system
  sensors.t1/t2/t3
  pressure
  flow
  outputs.htc1/htc2/antifreeze/alarm
  valves.upper/left/right
  alarms
  communication
  quality
```

快照带设备时间、上位机接收时间、采集周期号和数据质量；前端只消费领域快照，不直接消费寄存器数组。

### 5.6 `config_service.py`

配置写入是显式事务：

1. 读取 HR0..4，确认协议、暂存状态和当前代次。
2. 读取全部配置区，形成 `baseConfig`。
3. 用户编辑得到 `draftConfig`，后端计算字段级差异。
4. 按连续地址使用 FC16 写暂存；单点可用 FC06。
5. 回读被写区，确认暂存值一致。
6. 用户确认后写 `HR3=0xC6A6`。
7. 轮询 HR0..4，要求错误码为 0、提交成功位有效、代次递增。
8. 任一步失败时允许写 `HR3=0xD15C` 放弃，并重新读取。

串口地址、波特率、校验位属于高风险配置，单独成组，提交前提示可能导致当前连接失效；提交后按新参数重连并重新校验协议字。

### 5.7 `control_service.py`

即时命令不进入配置事务：

- HR800：远程加热。
- HR801..803：HTC1、HTC2、防冻的自动/强制关/强制开。
- HR804..806：上、左、右阀门，0 释放、1 原位、2 工作位。
- HR807：三路阀门故障复位掩码。
- HR808..816：阀门故障、控制源、远程租约剩余时间。

安全规则：

1. 强制开、阀门动作和故障复位必须二次确认。
2. 写入后立即读取 IR300..337 和 HR800..816 验证实际效果。
3. 远程阀门命令租约为 600 秒。用户选择“持续保持”时，每 60 秒刷新一次；停止采集、退出控制页或程序关闭时主动写 0 释放。
4. UI 同时显示“下发目标”和“实际状态”，禁止用按钮状态冒充设备反馈。
5. 控制源为安全或维护时，明确提示远程命令可能不生效。

### 5.8 `alarm_service.py`

读取 IR400..405，保存每个 `group + bit` 的发生和恢复事件。当前业务错误主要写入错误组 1，但三个组都必须读取和展示，避免后续固件扩展时丢失信息。

事件字段：设备、发生时间、恢复时间、组、位、错误枚举、中文说明、严重级别、相关对象、确认状态。

告警严重级别由上位机展示规则定义，不改变下位机错误位：

- 严重：阀门驱动、过流、开路、加热输出故障。
- 警告：传感器、阈值、回零/目标校验、开关卡滞。
- 信息：压力关闭状态等运行提示。

### 5.9 `sqlite_store.py`

建议数据表：

```text
devices
samples_environment
samples_outputs
samples_valves
alarm_events
control_events
config_transactions
communication_events
sessions
```

写盘使用单独队列和单写线程，采集线程不直接等待磁盘。原始 1 秒数据按保留天数清理；历史查询按时间跨度自动降采样。导出时生成新的、字段明确的 CSV，不再追求旧 CSV 正则兼容。

## 6. 轮询设计

| 组 | 区域 | 周期 | 说明 |
| --- | --- | --- | --- |
| 启动校验 | FC04 0..9 | 连接时 | 必须先确认 `0x0700` |
| 快速状态 | FC04 0..9 | 1 秒 | 总状态、RTC、主显示值 |
| 三路温湿度 | FC04 100..117 | 1 秒 | 三路独立值、状态、有效性 |
| 压力 | FC04 200..203 | 1 秒 | 压力及类型 |
| 流量 | FC04 210..213 | 1 秒 | 流量及呼吸状态 |
| 输出 | FC04 300..319 | 1 秒 | 实际状态、模式、计数 |
| 阀门 | FC04 320..337 | 1 秒 | 三路阀门完整诊断 |
| 告警 | FC04 400..405 | 1 秒 | 告警发生/恢复 |
| 通信 | FC04 500..506 | 5 秒 | 下位机侧通信健康 |
| 运行控制 | FC03 800..816 | 5 秒；控制时 1 秒 | 命令、故障和租约 |
| 配置 | FC03 各配置分区 | 连接后一次/手动 | 不做常规高频轮询 |

调度器应统计单轮耗时。如果同一串口设备过多导致 1 秒目标无法满足，UI 显示实际采样周期，不制造“实时”假象。

## 7. 页面与交互设计

### 7.1 设备中心

- 设备名称、COM 口、从站地址、波特率、校验位。
- 在线状态、协议版本、最后响应、实际周期、连续失败。
- 同一串口多设备关系可视化。
- 新增、编辑、删除、连接、断开、重连。
- 删除 TCP、旧设备类型和自定义旧点表选项。

### 7.2 实时总览

首页必须在一屏内回答“设备现在是否正常、正在做什么”：

- 顶部：设备、在线状态、设备 RTC、协议版本、活动告警数。
- 三路温湿度卡：左/T1、右/T2、上/T3，各自温度、湿度、状态、有效性。
- 过程量：压力、压力类型、流量、呼吸状态。
- 加热区：HTC1、HTC2、防冻、报警输出的实际状态和控制模式。
- 阀门区：上、左、右阀门位置、运动/故障、控制源、租约倒计时。
- 最近事件：告警、恢复、控制、配置提交、通信中断/恢复。

无效或过期数据必须灰显并带时间，不显示为正常的 0。

### 7.3 趋势曲线

曲线分为三个联动图组：

1. 三路温度和三路湿度，可独立隐藏、同类对比。
2. 压力和流量，使用独立坐标轴。
3. 状态带：HTC1、HTC2、防冻、呼吸状态、三路阀门、告警。

保留并优化：

- 滚轮缩放、框选、拖动、全屏、十字指针、联动 tooltip。
- 1 分钟、10 分钟、1 小时、24 小时和自定义范围。
- 阈值线、告警点、控制事件、配置提交事件叠加。
- 暂停自动滚动不停止后台采集。
- 当前窗口最小值、最大值、平均值和有效采样率。
- PNG 和 CSV 导出。

不再提供预测曲线、推荐区间和模拟结果叠加。

### 7.4 阀门与输出

独立工况页，避免关键控制埋在普通参数表里：

- 三路阀门使用一致的状态组件。
- 显示状态、执行状态、物理位置、故障、电流 ADC、控制源、租约。
- 提供回原位、到工作位、释放远程控制、清除故障。
- 输出提供自动、强制关、强制开。
- 每次操作显示“请求中—已响应—实际生效/未生效”，并写操作日志。

### 7.5 告警中心

- 当前告警与历史告警分栏。
- 可按设备、对象、严重级别、时间筛选。
- 阀门类告警直接链接到对应阀门详情。
- 传感器告警直接链接到对应趋势窗口。
- 用户确认只作用于上位机记录，不清除下位机错误位。

### 7.6 参数配置

按下位机结构分组：

1. 三路温湿度。
2. 压力与流量。
3. 阀门路由与通道。
4. 湿度、防冻和传感器故障动作。
5. 输出与告警。
6. 日志。
7. Modbus 通信。

页面固定显示：设备当前值、编辑值、差异、地址、类型、单位和校验错误。底部只有“写入暂存”“提交保存”“放弃修改”三种明确动作。

### 7.7 通信诊断

- 当前串口参数、请求速率、平均/最大响应时间。
- 上位机超时、CRC 错误、异常码、从站地址不匹配。
- 下位机 IR500..506 的在线状态和时间。
- 原始请求和响应都展示标准 RTU 帧及其字段解析。
- 仅提供单次、限长、带确认的诊断命令，不允许加入永久自动轮询。

### 7.8 历史与导出

- 按设备和会话查看历史趋势、告警、控制记录、配置事务。
- 支持导出所选时间段的环境、输出、阀门和事件 CSV。
- 不再展示旧“导入数据、运行日志分析、配置快照比较、客户报告”页面。

所有写接口必须带操作 ID，后端用它关联请求帧、响应帧、回读结果和最终状态，防止用户重复点击产生无法追踪的控制。
