# YLDQ 数据分析系统设计规划

## 1. 目标

这个系统不是再做一套单次离线脚本，而是做一个客户能直接打开、直接看懂、直接导出的本地分析台。

核心目标：

1. 一键打开。
2. 一页看到趋势、统计、异常和关键事件。
3. 用当前目录真实数据直接分析，不要求客户整理数据。
4. 后续可扩展到标准报告输出和多设备对比。

## 2. 当前数据资产

目前 `实时数据` 目录下至少包含三类有效数据：

1. `data_0\log_YYYY_MM_DD.csv`
   - 分钟级环境日志
   - 当前可解析字段：压力、温度、流量、湿度
2. `breath_data\breath_YYYY_MM_DD.csv`
   - 秒级呼吸变化日志
   - 当前可解析字段：时间、状态、流速、距上次变化时长、节律标记
3. `run\*.csv`
   - 运行过程日志
   - 当前可解析字段：时间、级别、消息
4. `config.json`
   - 当前配置快照
   - 可提供阈值、任务计划和运行参数

这意味着系统天然可以同时回答四类问题：

1. 设备环境变化趋势如何。
2. 呼吸相关状态是否正常。
3. 运行期间是否有错误、告警或异常切换。
4. 当前配置和实际数据是否匹配。

## 3. 首版架构

首版采用：

- `Python 本地服务`
- `HTML/CSS/JS 本地前端`
- `BAT 一键启动`

原因：

1. 可以直接读取本地文件夹，不受浏览器本地文件权限限制。
2. 不要求安装数据库或 Node 环境。
3. 可以把数据解析和统计逻辑留在服务端，后续更容易固化为正式产品。
4. 客户使用路径清晰，双击即可打开。

## 4. 功能模块规划

### 4.1 总览模块

输出：

- 数据覆盖时间
- 环境日志总条数
- 呼吸日志总条数
- 运行日志总条数
- 最新环境采样值
- 平均日完整率

价值：

- 客户打开第一页就知道数据是否完整、时间是否对、最近设备状态怎样。

### 4.2 自动诊断模块

首版用规则诊断输出重点问题，例如：

- 日完整率偏低
- 时间断档
- 阈值越界明显
- 运行日志错误过多
- 呼吸告警状态存在

价值：

- 客户不用先学图表，先看系统结论。

### 4.3 环境趋势模块

展示：

- 压力趋势
- 温度趋势
- 呼吸流速趋势
- 湿度趋势
- 对应高低阈值参考线

附加统计：

- 每日完整率
- 每个指标的均值、最小值、最大值
- 越界次数统计

价值：

- 快速定位哪一天、哪个量发生异常。

### 4.4 呼吸分析模块

展示：

- 呼吸流速时序
- 吸气/呼气/无呼吸/告警状态分布
- 节律标记分布
- 每日呼吸记录量
- 呼吸片段数量与最长持续时长

价值：

- 把原本只适合工程师看的原始呼吸 CSV，转换成客户可理解的状态视图。

### 4.5 运行日志模块

展示：

- 每日信息/警告/错误统计
- 关键词热点
- 关键事件时间线

重点识别方向：

- watchdog
- Flow / I2C
- LoRa
- IEC61850
- alarm
- open / close / heat control

价值：

- 让客户能把“曲线异常”跟“运行事件”对上时间。

### 4.6 配置快照模块

展示：

- 当前版本
- 温湿压阈值
- 呼吸阈值
- 无变化告警时长
- 双路模式
- 强制关闭时间
- 任务计划表

价值：

- 回答“这是在什么配置下采出来的数据”。

### 4.7 数据质量模块

展示：

- 重复时间戳
- 断档数
- 最长断档
- 解析失败行数
- 每日日完整率偏低记录

价值：

- 客户和售后能先判断是“设备异常”还是“数据本身不完整”。

## 5. 首版实现边界

这次落地的是“可运行原型”，重点是把主干能力搭起来。

已做：

1. 本地服务可直接分析真实目录数据。
2. 页面已具备客户可用的统一仪表板。
3. 已支持导出 JSON、文本简报、打印页面。

暂未做：

1. 多设备横向对比。
2. 固定模板 PDF/Word 正式报告。
3. 按日期区间筛选。
4. 单日深钻页面。
5. 更细的规则引擎和告警归因。

## 7. 当前结论

最合适的方向不是继续堆脚本，而是把现有脚本能力收口成一个本地分析台。

这样做的好处是：

1. 对客户，交互简单。
2. 对你后续维护，结构清晰。
3. 对未来扩展到正式产品，路径连续。
