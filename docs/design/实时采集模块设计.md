# YLDQ 实时采集模块设计

## 1. 目标

在当前 `YLDQ 数据分析系统` 内增加一个 `实时采集` 模块，用于直接通过下位机 Modbus 通讯实时采集、显示、控制和保存数据。

设计原则：

1. 以协议文档和下位机真实代码映射为准，不以旧 FUXA 画面为准。
2. 采集、显示、改参、导出、立即分析在同一程序内闭环完成。
3. 采集结果直接写成当前分析系统兼容目录格式，做到“采了立马分析”。
4. 多设备可配置、可切换，且每个设备具备独立通讯参数。
5. 协议可写的参数才允许编辑，并且写入后必须回读确认。

- [dashboard_server.py](</C:/Users/pc/Desktop/4.0.1-2/xishiqi4.0.1/docs/新程序运行数据分析系统/dashboard_server.py>)

## 2. 设计边界

### 2.1 真实数据源优先级

实时采集模块只认两层真相：

1. 协议文档中明确开放的功能码、地址和数据类型
2. 固件实际映射和写入逻辑

当两者冲突时，以固件当前实现为准，并在点表中标记“协议文档与实现差异”。

### 2.2 当前可用协议范围

根据协议文档和固件实现，首版支持：

1. `0x01` 读线圈状态
2. `0x02` 读离散输入状态
3. `0x03` 读保持寄存器
4. `0x04` 读输入寄存器
5. `0x05` 写单个线圈
6. `0x06` 写单个寄存器
7. `0x0F` 写多个线圈
8. `0x10` 写多个寄存器

固件写入入口见 [modbusModule.c](</C:/Users/pc/Desktop/4.0.1-2/xishiqi4.0.1/SerialLinker/protocols/modbusModule.c:163>)。

### 2.3 当前做不到的部分

以下内容必须明确不是首版“协议直采能力”：

1. **固件内部运行日志**
   - 当前 Modbus 协议未提供日志流读取接口。
   - 因此无法仅靠实时 Modbus 采集生成与 SD 卡 `run/*.csv` 完全等价的固件运行日志。

2. **完整原始呼吸状态枚举**
   - 当前离散输入只提供吸气/呼气等布尔态。
   - 固件里 `ALARM_LOW` 也归到呼气位，`ALARM_HIGH` 归到吸气位。
   - 因此上位机可推导“方向态”，但无法 100% 还原底层完整状态枚举。

3. **所有配置项全量可写**
   - 首版只开放协议已映射、固件已处理、且明确支持写入的项目。
   - 其余配置在页面中显示为只读。

## 3. 与现有分析系统的关系

当前分析系统是离线/本地数据分析台，读取：

1. `实时数据/config.json`
2. `实时数据/data_0/log_*.csv`
3. `实时数据/breath_data/breath_*.csv`
4. `实时数据/run/*.csv`

实时采集模块不是替代现有分析系统，而是新增一个上游数据源模块：

1. 通过 Modbus 实时采集下位机数据
2. 写入与当前分析系统兼容的目录格式
3. 用户点击“立即分析当前会话”后，直接复用现有分析页面与算法

这意味着：

1. 不重写已有分析算法
2. 不废弃总体曲线、呼吸事件、配置快照、参数推荐等现有页面
3. 新模块重点解决“如何实时采到一份可分析的数据资产”

## 4. 用户场景

### 场景 1：单设备实时查看

1. 用户配置一个设备：串口、地址、波特率
2. 点击开始采集
3. 在 `实时曲线` 页面查看温湿压流与状态变化
4. 在 `参数控制` 页面修改阈值或控制位
5. 保存为一个采集会话
6. 直接切换到分析页查看总体曲线和推荐值

### 场景 2：多设备切换

1. 用户提前配置多个设备档案
2. 左侧设备中心展示在线状态
3. 主视图只聚焦当前选中设备
4. 可一键切换设备

### 场景 3：售后现场调试

1. 连接设备后立即查看实时流量和加热状态
2. 修改阈值并回读确认
3. 导出本次会话目录
4. 把会话目录直接交给当前分析系统继续分析

## 5. 导航结构

新增一级导航页：`实时采集`

子页建议拆为 5 个：

1. `设备中心`
2. `实时总览`
3. `实时曲线`
4. `参数控制`
5. `会话导出与分析`

### 5.1 设备中心

负责设备档案、通讯配置、在线状态、切换和重连。

### 5.2 实时总览

负责显示当前值、报警、连接状态、最近写入结果、统计卡片。

### 5.3 实时曲线

借鉴当前 `总体曲线` 的交互能力，显示连续曲线、状态带和事件点。

### 5.4 参数控制

展示当前设备所有可写参数和控制项，支持编辑、写入、回读、失败提示。

### 5.5 会话导出与分析

显示当前会话目录、写盘状态、兼容性状态，并提供“立即分析当前会话”入口。

## 8. 派生点设计

为了让实时采集结果可直接用于现有分析系统，需要定义一批派生点。

### 8.1 通讯健康类

1. 在线状态
2. 最后成功读取时间
3. 最后成功写入时间
4. 连续超时次数
5. CRC 失败次数
6. 地址不匹配次数
7. 最近错误文本

### 8.2 曲线分析类

1. 最近 1 分钟最大值/最小值/均值
2. 越阈值次数
3. 最近状态切换次数
4. 最近报警次数

### 8.3 呼吸派生类

1. 方向态：吸气/呼气/未知
2. 流速变化段
3. 上次状态切换距今时长
4. 节律标记

这些派生值在 `breath_data/*.csv` 中落盘时要标记来源，避免误认为来自原始固件日志。

7. 通道二加热状态
8. 辅热状态
9. 阀门状态

### 9.2 标准组，5 秒

1. 温度
2. 湿度

  breath_data/
    breath_2026_05_22.csv
  run/
    acquisition_2026_05_22.csv

## 16. 风险与验证点

### 16.1 协议与固件不一致风险

验证方式：

1. 用实际设备逐地址读写核对
2. 固件点表与协议文档逐项比对
3. UI 中显式标记异常点

### 16.2 实时曲线性能风险

验证方式：

1. 限制内存缓存窗口
2. 长会话做抽样绘制
3. 全屏与缩放交互测试

### 16.3 会话兼容性风险

验证方式：

1. 把实时采集生成的目录直接喂给现有 `/api/analysis`
2. 确认总览、总体曲线、呼吸页、配置快照都能读

### 16.4 写参安全风险

验证方式：

1. 所有写入都先验证类型和范围
2. 所有写入都做回读确认
3. 关键控制位单独弹确认

## 17. 当前实施结论

这不是再造一个新上位机，而是在当前分析系统里新增一个协议驱动的数据入口层。

正确路线是：

1. 先完成点表和设备模型
2. 再做实时读取
3. 再做曲线和参数控制
4. 最后接入现有分析链路

只有这样，才能同时满足：

1. 多设备切换
2. 全量可视化
3. 协议和固件一致
4. 改参可回读
5. 实时采集后立即分析
